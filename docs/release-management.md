# Release Management

QuickBite uses GitHub Actions to make releases repeatable without locking the project into a specific hosting platform yet.

## Release Workflow

The `.github/workflows/release.yml` workflow runs on:

- pushes to `main`
- semantic release tags like `v1.2.3`
- manual `workflow_dispatch` promotions

The workflow performs these stages:

- Validate: restore, build, backend tests, frontend Vitest tests, Playwright smoke tests, frontend production build, and Docker Compose config validation.
- Dependency scan: generate NuGet vulnerability and npm audit reports as release artifacts.
- Migration bundles: create one EF Core migration bundle per service database.
- Container images: build and publish one image per API, gateway, and web app to GitHub Container Registry.
- Image scan: scan published images with Trivy and upload SARIF results.
- Promotion: record staging promotion from `main`, and production promotion from release tags or explicit manual selection.

## Image Registry

Default registry: GitHub Container Registry.

Images are published under:

```text
ghcr.io/<owner>/quickbite-identity-api
ghcr.io/<owner>/quickbite-catalog-api
ghcr.io/<owner>/quickbite-orders-api
ghcr.io/<owner>/quickbite-payments-api
ghcr.io/<owner>/quickbite-delivery-api
ghcr.io/<owner>/quickbite-gateway
ghcr.io/<owner>/quickbite-web
```

Each image receives immutable `git-<commit-sha>` tags. Branch, release-tag, and `latest` tags are also generated by Docker metadata action. Rollbacks should use immutable `git-<commit-sha>` tags, not mutable `latest`.

## Migration Sequencing

QuickBite has one database per service, so deployments must apply migrations service-by-service before rolling the corresponding service image.

Recommended order:

1. Identity
2. Catalog
3. Orders
4. Payments
5. Delivery

The release workflow uploads migration bundles as artifacts. Operators should run each bundle with that service database connection string before deploying the matching service container. This keeps schema changes explicit and avoids hidden startup-time migration behavior in production-like environments.

Local bundle generation:

```powershell
./scripts/build-migration-bundles.ps1
```

## Promotion Strategy

Staging promotion happens after images and migration bundles are produced from `main`. Production promotion requires either:

- a semantic version tag such as `v0.1.0`
- a manual workflow dispatch selecting `production`

GitHub Environments should be configured with reviewers/secrets when real hosting exists. Until hosting is chosen, the workflow records promotion intent and preserves artifacts without executing provider-specific deployment commands.

## Rollback Strategy

Rollback should use the last known-good `git-<commit-sha>` image set and the matching migration plan.

Safe rollback expectations:

- Prefer backward-compatible migrations.
- Avoid destructive schema changes in the same release as application changes.
- Keep release notes with the image SHA set and migration bundle artifacts.
- If a migration is not backward-compatible, prepare a compensating migration and rehearse it in staging first.

## Release Tagging

Use semantic version tags:

```text
v0.1.0
v0.2.0
v1.0.0
```

Tag only commits already merged to `main`. The tag should represent the exact image set and migration bundles promoted to production.
